- **用途**：参考游戏设计、查看功能需求
- **PGB1展示**：可能单独展示或作为辅助视图

#### 01_Preproduction - 预制作（原画组）
- **内容**：
  - 原画组提供的设计稿（主要）
  - 原画制作时使用或参考的素材（未来可能有）
- **文件类型**：.psd, .psb（主要），未来可能有其他参考素材
- **用途**：设计参考、素材源文件
- **PGB1展示**：与02/04/05合并展示（辅助素材视图）

#### 02_Production - 制作（Spine组）
- **内容**：Spine组制作的内容（一般没有，有时会有）
- **文件类型**：序列帧内容
- **用途**：Spine动画相关
- **PGB1展示**：与01/04/05合并展示（辅助素材视图）

#### 03_Render_VFX - ⭐ 渲染和特效（主战场）
- **内容**：特效师（用户）制作的所有素材
- **重要性**：**程序要整理的内容基本都在这里**
- **结构**：见下方详细章节
- **PGB1展示**：**主视图**，双视图模式（树形分类 + 名称排序）

#### 04_Trailer - 宣传片素材
- **内容**：基本没用过，大部分项目为空
- **PGB1展示**：与01/02/05合并展示（辅助素材视图）

#### 05_Outside - 外部素材收集
- **内容**：特效师搜集的制作素材（参考、资源）
- **用途**：制作时的参考和资源库
- **PGB1展示**：与01/02/04合并展示（辅助素材视图）

## 🎯 核心工作目录结构（03_Render_VFX/VFX/）

```
03_Render_VFX/VFX/
├── AE/                                # AE工程文件存放地
├── Export/                            # AE输出的内容（主要工作目录）
│   ├── {功能分类1}/                    # 由用户表单生成，不固定
│   ├── {功能分类2}/                    # 每个项目可能不同
│   └── ...
├── nextcloud/                         # 本地网盘同步目录
│   └── （结构待详细讨论）
└── PSD/                               # 拆分后的PSD文件
    ├── {功能分类1}/                    # 至少同步表单中的分类
    ├── {功能分类2}/                    # 可能有用户自建的额外文件夹
    └── ...
```

### 各子目录详细说明

#### AE/ - AE工程文件
- **内容**：After Effects工程文件（.aep）
- **用途**：制作特效的源文件
- **PGB1处理**：简单列表展示，可能需要快速打开功能

#### Export/ - ⭐ AE输出内容（主要工作目录）
- **内容**：从AE导出的所有素材
- **重要特性**：**下级分类不是固定的**
- **分类生成**：
  - 用户在程序内填写表单
  - 程序根据表单自动生成格式化的文件夹结构
  - 每个项目的分类可能不同
- **示例**（217_RedDevil项目的分类）：
  - Ambient（环境特效）
  - Free Spin（免费旋转）
  - Infoboard（信息板）
  - Main UI（主界面）
  - Mouse Hover（鼠标悬停）
  - Prototype（原型）
  - Spin Button（旋转按钮）
  - Win Highlight（赢得高亮）

#### nextcloud/ - 本地网盘同步目录

**核心概念**：nextcloud是 `02_done/` 的简化副本，用于标记已上传状态。

**目录结构**：
```
nextcloud/
├── {功能分类1}/          # 按功能分类（同Export下的分类）
│   ├── 文件1.webp        # 扁平化存储，无子文件夹
│   ├── 文件2.plist
│   ├── 文件2-0.webp
│   └── ...
├── {功能分类2}/
└── ...
```

**复制规则**：
1. **来源**：`02_done/` 中整理后的文件
2. **排除**：TexturePacker工程文件（`.tps`）不复制
3. **包含**：
   - 静帧图片（`.webp`等）
   - 精灵图文件（`.webp`）
   - 配置文件（`.plist`）
4. **结构简化**：
   - ❌ 不保留done的文件夹结构（`[img-XX]/`, `[an-XX-YY]/`）
   - ✅ 按功能分类扁平化存储

**示例对比**：

**02_done结构**（复杂分类）：
```
Export/Ambient/02_done/
├── [img-100]/
│   ├── main_vfx_a.webp
│   └── main_vfx_g.webp
├── [img-50]/
│   └── main_vfx_b.webp
└── [an-50-18]/
    ├── bonus_vfx_a.plist
    ├── bonus_vfx_a.tps        ← 不复制到nextcloud
    └── bonus_vfx_a-0.webp
```

**nextcloud结构**（扁平化）：
```
nextcloud/Ambient/
├── main_vfx_a.webp          ← 从[img-100]复制
├── main_vfx_g.webp          ← 从[img-100]复制
├── main_vfx_b.webp          ← 从[img-50]复制
├── bonus_vfx_a.plist        ← 从[an-50-18]复制
└── bonus_vfx_a-0.webp       ← 从[an-50-18]复制
（没有.tps文件，所有文件在同一层级）
```

**重要功能**：

1. **反向标记已上传状态**：
   - ⚠️ **不能立即复制**：用户上传到网盘后，才在程序内标记"已上传"，然后复制到nextcloud
   - ✅ 如果文件存在于nextcloud，说明已上传
   - ❌ 如果文件不存在于nextcloud，说明未上传或上传失败

2. **本地副本**：
   - 与网盘内容保持同步
   - 可以快速对比本地和网盘的差异

3. **上传检查**：
   - 在拖拽上传前，检查nextcloud中是否已存在
   - 避免重复上传

#### PSD/ - 拆分后的PSD文件
- **内容**：从原画组拿到PSD后拆分导出的素材
- **用途**：导入AE制作特效
- **文件夹结构**：
  - **必须包含**：表单中定义的所有功能分类
  - **可能额外有**：用户自建的其他文件夹
- **命名规则**：比较随意，不强制格式
- **PGB1要求**：
  - 至少自动创建表单中的分类文件夹
  - 允许用户手动新建额外文件夹
  - 在程序界面中反映所有文件夹（包括自建的）

## 📝 项目表单系统（核心功能）

### 概念
- 每个项目开始时，用户在PGB1程序内**选择任务**
- 任务定义该项目的**功能分类列表**（不是固定的，每个项目可能不同）
- 程序根据选择的任务自动生成标准化的文件夹结构

### 核心架构：全局任务清单 + 项目任务启用

#### 全局任务清单
- **存储位置**：跟随软件安装，所有项目共享
- **用户管理**：
  - ✅ 可以手动添加新任务
  - ✅ 可以重命名任务
  - ✅ 可以删除任务（有保护机制）
- **同步规则**：任务清单本身在所有项目间同步（公共资源池）

#### 项目任务启用（每个项目独立）
- **独立配置**：每个项目记录自己启用的任务列表
- **选择方式**：从全局任务清单中勾选需要的任务
- **生成规则**：开启任务后，程序自动创建对应的文件夹结构

### 任务清单管理

#### 添加新任务
- **触发**：用户在任务管理界面点击"添加任务"
- **输入**：任务名称（如"New Feature"）
- **结果**：任务添加到全局清单，所有项目可见

#### 重命名任务
- **触发**：用户在任务管理界面选择任务并重命名
- **范围**：全局生效，所有项目中该任务的名称同步更新
- **注意**：已经创建的文件夹名称可能需要手动调整（或程序提供重命名工具）

#### 删除任务（有保护机制）
- **检查机制**：删除前检查该任务是否在任何项目中被启用
- **如果有项目在使用**：
  - ❌ 阻止删除
  - 提示用户："任务 'Ambient' 正在被以下项目使用：[项目A, 项目B]。无法删除。"
- **如果没有项目使用**：
  - ✅ 允许删除
  - 任务从全局清单中移除

### 项目中关闭任务（归档机制）

#### 触发场景
用户在某个项目中不再需要某个任务，想要关闭它。

#### 警告确认
- **弹出提示**："确定要关闭任务 'Ambient' 吗？该任务的所有文件将被归档。"
- **选项**：确认 / 取消

#### 归档处理（不真的删除）
**执行操作**：
1. 将任务对应的文件夹移动到项目的 `.archived_tasks/` 目录
2. 更新项目配置，标记该任务为"已关闭"
3. 在任务管理界面中，该任务的开关变为关闭状态

**归档结构示例**：
```
项目文件夹/
├── Export/
│   ├── Free Spin/
│   └── Infoboard/
├── PSD/
├── nextcloud/
└── .archived_tasks/           # 归档目录（隐藏）
    └── Ambient/                # 被关闭的任务
        ├── timestamp_2026-02-13_14-30/  # 第一次归档（时间戳）
        │   ├── Export/
        │   ├── PSD/
        │   └── nextcloud/
        └── timestamp_2026-02-14_10-15/  # 第二次归档（时间戳）
            ├── Export/
            ├── PSD/
            └── nextcloud/
```

**关键设计**：
- ✅ 使用时间戳区分多次归档
- ✅ 保留完整的目录结构
- ✅ 用户可以手动恢复文件（或使用"时光机"功能）

### 时光机功能（恢复归档任务）

#### 功能定位
- 用户可以查看已归档的任务
- 选择特定时间的归档版本进行恢复

#### UI界面
```
归档任务列表：
┌─────────────────────────────────────────┐
│ Ambient                                 │
│   ├─ 2026-02-14 10:15  [恢复] [删除]   │
│   └─ 2026-02-13 14:30  [恢复] [删除]   │
│ Main UI                                 │
│   └─ 2026-02-10 09:00  [恢复] [删除]   │
└─────────────────────────────────────────┘
```

#### 恢复操作
1. 用户选择要恢复的归档版本
2. 程序检查当前是否有同名任务已启用
   - 如果有冲突：提示用户选择"合并"或"覆盖"
   - 如果无冲突：直接恢复
3. 将归档文件夹移回正常位置
4. 更新项目配置，标记任务为"已启用"

#### 彻底删除归档
- 用户可以在时光机界面中彻底删除某个归档版本
- 操作不可逆，需要二次确认

### 自动生成的目录结构总结

根据任务，程序会在以下位置自动创建对应的文件夹：
1. `Export/{任务名称}/` - 每个任务下创建标准工作流子目录（00/01/02/03）
2. `PSD/{任务名称}/` - 至少创建表单中的任务（用户可能额外新建）
3. `nextcloud/{任务名称}/` - 用于标记已上传状态
4. `.archived_tasks/{任务名称}/timestamp_YYYY-MM-DD_HH-MM/` - 归档任务的历史版本

**特殊处理**：
- Prototype任务：在00/01/02下额外创建7个固定子分类，nextcloud保留子分类结构并添加_original目录

### 技术实现要点

#### 数据存储
- **全局任务清单**：存储在软件配置文件（如 `~/.pgb1/global_tasks.json`）
- **项目任务配置**：存储在项目配置文件（如 `项目文件夹/.pgb1_project.json`）

**示例结构**：
```json
// global_tasks.json
{
  "tasks": [
    {"id": "ambient", "name": "Ambient", "type": "normal"},
    {"id": "free_spin", "name": "Free Spin", "type": "normal"},
    {"id": "prototype", "name": "Prototype", "type": "special"}
  ]
}

// .pgb1_project.json
{
  "project_name": "217_RedDevil",
  "enabled_tasks": ["free_spin", "infoboard", "prototype"],
  "archived_tasks": [
    {
      "task_id": "ambient",
      "archived_at": "2026-02-13T14:30:00",
      "path": ".archived_tasks/Ambient/timestamp_2026-02-13_14-30/"
    }
  ]
}
```

#### 任务删除检查逻辑
```python
def can_delete_task(task_id):
    # 扫描所有项目的配置文件
    for project in all_projects:
        if task_id in project.enabled_tasks:
            return False, project.name
    return True, None
```

#### 归档时间戳格式
```python
import datetime
timestamp = datetime.now().strftime("timestamp_%Y-%m-%d_%H-%M")
# 结果: "timestamp_2026-02-13_14-30"
```

## 📂 Export工作流（四级子目录）

每个功能分类下都有统一的4个子目录，对应完整的制作工作流：

```
Export/{功能分类}/
├── 00_original/          # ⚙️ AE原始导出
├── 01_scale/             # 🔄 缩放处理

│   ├── [img-70]/         # 静帧，70%
│   ├── [img-50]/         # 静帧，50%
│   ├── [an-100-24]/      # 序列帧（精灵图），100%，24帧
│   ├── [an-50-18]/       # 序列帧（精灵图），50%，18帧
│   └── [an-40-18]/       # 序列帧（精灵图），40%，18帧
```

#### 技术要点

**1. .tps 文件 scale 解析**

.tps 文件是XML格式，需要解析 `<scale>` 节点：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<data>
    <struct type="Settings">
        <key>scale</key>
        <double>0.7</double>  <!-- 这里是最终的scale值 -->
        ...
    </struct>
</data>
```

解析逻辑：
```javascript
async function parseTpsScale(tpsPath) {
  const content = await readFile(tpsPath, 'utf-8');
  const match = content.match(/<key>scale<\/key>\s*<double>([\d.]+)<\/double>/);
  if (match) {
    const scaleDecimal = parseFloat(match[1]);  // 如 0.7
    const scalePercent = Math.round(scaleDecimal * 100);  // 转为 70
    return scalePercent;
  }
  throw new Error(`无法从 ${tpsPath} 解析 scale 值`);
}
```

**2. 三件套完整性检测**

移动文件前检查三个文件是否都存在：
```javascript
async function validateThreePack(outputDir, name) {
  const files = [
    `${outputDir}/${name}.webp`,
    `${outputDir}/${name}.plist`,
    `${outputDir}/${name}.tps`
  ];
  for (const file of files) {
    if (!await fileExists(file)) {
      throw new Error(`缺少文件: ${file}`);
    }
  }
  return true;
}
```

**5. 窗口管理（置前逻辑）**

因为转换过程中会打开多个窗口（imagine、多个TexturePacker GUI），需要自动将这些窗口置于前面，方便用户操作。

**优先级**：imagine > TexturePacker GUI

**技术实现**（Windows API）：

```rust
use windows::Win32::Foundation::*;
use windows::Win32::UI::WindowsAndMessaging::*;

// 等待窗口创建（轮询检测，最多2秒）
async fn wait_for_window(pid: u32) -> Result<HWND> {
    let timeout = Duration::from_secs(2);
    let start = Instant::now();

    loop {
        if let Some(hwnd) = find_window_by_pid(pid) {
            return Ok(hwnd);
        }

        if start.elapsed() > timeout {
            return Err("窗口创建超时");
        }

        tokio::time::sleep(Duration::from_millis(100)).await;
    }
}

// 根据进程ID查找主窗口
fn find_window_by_pid(pid: u32) -> Option<HWND> {
    let mut result = None;

    unsafe {
        EnumWindows(Some(enum_windows_callback), LPARAM(&mut result as *mut _ as isize));
    }

    result
}

// 将窗口置于最前面
async fn bring_window_to_front(pid: u32) {
    if let Ok(hwnd) = wait_for_window(pid).await {
        unsafe {
            // 1. 恢复窗口（如果最小化）
            ShowWindow(hwnd, SW_RESTORE);

            // 2. 置顶窗口（临时）
            SetWindowPos(
                hwnd,
                HWND_TOPMOST,
                0, 0, 0, 0,
                SWP_NOMOVE | SWP_NOSIZE
            );

            // 3. 取消置顶但保持在最前（避免永久置顶）
            SetWindowPos(
                hwnd,
                HWND_NOTOPMOST,
                0, 0, 0, 0,
                SWP_NOMOVE | SWP_NOSIZE | SWP_SHOWWINDOW
            );

            // 4. 激活窗口
            SetForegroundWindow(hwnd);
        }
    }
}
```

**调用时机**：

1. **imagine窗口**：
   - 启动imagine后立即调用 `bringWindowToFront(imaginePid)`
   - 因为imagine优先级最高，且只有一个实例

2. **TexturePacker GUI窗口**（逐个处理）：
   - 每次启动新的TexturePacker GUI后调用 `bringWindowToFront(guiPid)`
   - 因为是逐个处理，所以每次只有一个GUI在前面
   - 用户关闭当前GUI后，下一个GUI自动打开并置前

**注意事项**：
- 不使用永久置顶（`HWND_TOPMOST`），避免遮挡悬浮窗和其他重要窗口
- 使用临时置顶后立即取消，只是将窗口移到Z-order顶部
- `SetForegroundWindow` 可能受Windows安全限制，需要程序有焦点时才能生效

## 🏷️ 文件命名规则

### 核心概念

#### 基础名称的定义

**静帧**：
```
文件名: main_vfx_a_add_seed_01.jpg
基础名: main_vfx_a_add_seed
规则: 去掉末尾的"_01"
```

**序列帧**：
```
文件名: bonus_vfx_a_screen_01.png
        bonus_vfx_a_screen_02.png
        ...
基础名: bonus_vfx_a_screen
规则: 去掉末尾的"_01"、"_02"等帧编号
```

**重要规则**：
- ✅ 基础名称 = 去掉末尾的 `_01` 或 `_NN`（帧编号）
- ✅ 后缀名（扩展名）不参与判断（因为会变化：jpg→png→webp）
- ✅ 判断同一素材：只需基础名完整匹配即可
- ✅ 序列帧不会有三位数的情况（01-99足够）

---

### 文件命名示例格式

#### 典型格式（含vfx标识）
```
{场景前缀}_vfx_{字母编号}_{混合模式}[_{补充说明}]
```

**示例**：
- `main_vfx_a_add_seed` - 主界面场景，特效a，add混合模式，种子元素
- `bonus_vfx_a_screen` - 奖励场景，特效a，screen混合模式
- `fs_vfx_c_screen` - 免费旋转，特效c，screen混合模式
- `sb_vfx_a_screen` - 旋转按钮，特效a，screen混合模式

**说明**：
- 实际使用时，特效师可以自由命名，不强制遵守此格式
- 但大部分文件会包含 `_vfx_` 标识
- 程序通过识别 `_vfx_` 来归类和排序

---

### 名称排序逻辑（名称排序视图）

#### 排序规则
在"名称排序视图"中，需要特殊的排序逻辑：

**步骤1：提取带有 `_vfx_` 的素材**
- 扫描所有文件的基础名
- 筛选出包含 `_vfx_` 字样的素材
- 这些归为一类显示

**步骤2：按 `_vfx_` 之后的内容排序**
- 提取 `_vfx_` 之后的部分
- 按字母（a, b, c...）排序
- 如果后面还有数字或字母，依次排序

**示例**：
```
原始基础名列表：
- main_vfx_a_add_seed
- main_vfx_b_normal
- main_vfx_c_screen
- main_vfx_a_add_a
- main_vfx_a_add_b

排序后（按_vfx_后的内容）：
- main_vfx_a_add_a      ← a_add_a
- main_vfx_a_add_b      ← a_add_b
- main_vfx_a_add_seed   ← a_add_seed
- main_vfx_b_normal     ← b_normal
- main_vfx_c_screen     ← c_screen
```

**Windows式自然排序**：
- 如果超过z，可能会出现 `aa`, `ab`, `ac` 等
- 程序直接使用类似Windows的自然排序算法即可
- 不需要特殊处理字母编号上限

**不含 `_vfx_` 的文件**：
- 正常按文件名字母顺序排序
- 可能单独分组或混合在列表中

---

### 序列帧命名
```
{基础名称}_{帧编号}.png
```

**示例**：
- `bonus_vfx_a_screen_01.png`
- `bonus_vfx_a_screen_02.png`
- `bonus_vfx_a_screen_18.png`

**规则**：
- 帧编号从 `01` 开始
- 补齐两位数（01, 02, ..., 99）
- 不会出现三位数（001, 002...）
- 扩展名通常为 `.png`（序列帧）

### 程序实现建议

**1. 提取基础名函数**
```python
def extract_base_name(filename):
    # 去掉扩展名
    name_without_ext = remove_extension(filename)
    # 去掉末尾的 _01, _02 等
    if name_without_ext.endswith("_01") or matches_pattern(name_without_ext, r"_\d{2}$"):
        base_name = remove_last_underscore_number(name_without_ext)
    else:
        base_name = name_without_ext
    return base_name
```

**2. 匹配同一素材**
```python
def is_same_asset(file1, file2):
    base1 = extract_base_name(file1)
    base2 = extract_base_name(file2)
    return base1 == base2  # 只比较基础名，忽略扩展名
```

**3. 名称排序视图**
```python
def sort_by_vfx_order(assets):
    # 提取带有 _vfx_ 的素材
    vfx_assets = [a for a in assets if "_vfx_" in a.base_name]

    # 提取 _vfx_ 之后的部分作为排序键
    def get_sort_key(asset):
        parts = asset.base_name.split("_vfx_")
        if len(parts) > 1:
            return parts[1]  # 返回 _vfx_ 之后的部分
        return asset.base_name

    # 使用自然排序（类似Windows）
    return natural_sort(vfx_assets, key=get_sort_key)
```

# PGB1 代码审查报告

> 本文档基于深入的静态扫描和逻辑排查生成，重点聚焦项目中亟待优化的技术债与低熵隐患。包括不同步的文档、巨型组件、不规范的 Rust 异常处理等。

## 1. 文档与代码不同步问题

### 1-1. `CODE_INDEX.md` 漏登新视图页面
- **现象**：`CODE_INDEX.md` 索引文件中，`src/views/` 目录下实际有 12 个页面文件，但统计数据显示为 9，且罗列时漏掉了十分核心的 `SettingsPage.vue` 组件（约 980 行）。
- **优化建议**：更新 `CODE_INDEX.md`，将 `SettingsPage.vue` 及其职责写入其中。

## 2. 遗留隐患与死代码清理

### 2-1. 残留的 Console Debug 日志
- **文件**：`src/components/TaskCard.vue`
- **处理结果**：此前存在的一句针对进度计算的临时验证调试代码 `console.log('[TaskCard] ...')`，已在此次排查期间直接进行了修复与删除处理。由于其随列表卡片循环高频执行，属于前端内存及日志污染项。

## 3. 架构重负与复杂逻辑解耦 (God Object)

### 3-1. Vue 巨型组件拆分迫在眉睫
- **文件**：`src/views/TaskPage.vue` (总计 ~2168 行)
- **现象**：`TaskPage.vue` 承接了极其复杂的职责。它囊括了双视图（名称/树形）的渲染逻辑、多选/框选机制、基于事件系统的拖拽与数据更新、以及版本处理和侧边栏的多入口等。单体组件维护成本极高，阅读难度大。
- **优化建议**：按功能或视图切分多个子组件：
  1. 将树形视图与列表视图拆分为 `TaskTreeList.vue` / `TaskFlatList.vue`
  2. 针对页面的业务流，提取新的 Composable（如 `useTaskUpload` 处理预览上传等独立逻辑）。

### 3-2. Composable 逻辑越界与过载
- **文件**：`src/composables/useStatusBar.ts` (总计 ~445 行)
- **现象**：该 Hook 除了负责状态栏的倒计时状态，还承接了非常底层的 IP 属地检测（`ipapi.co`）、不同国家的节假日 API 请求（`Timor.tech`, `date.nager.at`）、番茄钟的计时状态机（Pomodoro Phase）、并维护后台秒级 `tick`。
- **优化建议**：按领域（Domain）拆分：
  - `usePomodoro.ts` (专门维护番茄钟状态和相关提醒)
  - `useHolidayAPI.ts` (负责 IP 检测和节假日判定)
  - 保留干净的 `useStatusBar.ts` 作为上层组装 Hook。

## 4. 后端稳定性与规范风险

### 4-1. 致命的 `.unwrap()` 后端崩溃风险
- **文件**：`src-tauri/src/commands.rs` (第 641, 1948, 1954 行等) 以及 `lib.rs` (第 91, 99 行)。
- **现象**：后端有多处在操作路径读取或 `parent()` 层级查找时直接使用了 `.unwrap()`。根据防御性编程原则：如果目标结构意外变动或系统找不到指定部分，`.unwrap()` 将引发整个 Rust Tauri 后台进程直接宕机（Panic 退出）。
- **优化建议**：
  改为使用 `Result` 和 `?` 进行传播，或者利用 `if let Some(parent) = old_path.parent()` 等模式安全匹配异常流并用自定义 Error DTO 回传给前端渲染。

### 4-2. 异常捕获缺失正规的 Logger
- **文件**：`src-tauri/src/hotkey.rs`, `conversion.rs`, `commands.rs`。
- **现象**：捕获到了错误信息后，当前实现使用了 `eprintln!("PSD 解析失败 {}: {}", path, e);` 抛出警告。对于桌面级生产应用（最终形态是 `.exe` 且终端隐藏），该报错将“静默失败”，难以被回溯排查。
- **优化建议**：
  摒弃所有的 `println!` / `eprintln!`。引入 `tracing` 或者官方建议的 `tauri-plugin-log`，确保异常日志会持续记录在用户的 AppData 目录，便于崩溃追踪。

### 1. 卡片设计讨论

#### 背景
产品总监提供了四种卡片的参考设计图：
- `design/Image Example/卡片_项目.png`
- `design/Image Example/卡片_任务.png`
- `design/Image Example/卡片_素材.png`
- `design/Image Example/卡片_普通.png`

#### 讨论要点

**1.1 卡片类型与使用场景**
- **项目卡片**：主页的项目列表，点击进入项目页
- **任务卡片**：项目页的任务列表，点击进入任务页
- **素材卡片**：任务页的双视图展示，点击打开详情页
- **普通卡片**：辅助页面（游戏介绍、项目素材），点击打开详情页

**1.2 任务卡片的"状态 1/3"**
- **含义**：子任务系统，1=已完成的子任务数，3=总子任务数
- **背景颜色**：
  - 未开始（0/Y）：灰色
  - 进行中（0 < X < Y）：黄色
  - 已完成（X = Y）：绿色

**1.3 预览图生成规则**

| 文件类型 | 预览方式 |
|---------|---------|
| 图片类 | 显示图片内容 |
| 序列帧 | 显示缩略图动画 |
| 精灵图 | 沿用对应序列帧的缩略图动画 |
| 视频 | 显示视频内容 |
| 文档等其他 | 通用图标 |

**1.4 项目卡片的ICON**
- 来源：`01_Preproduction/` 中带"AppIcon"字样的PSD缩略图
- 备选：无则使用默认图标

**1.5 交互行为**
- **悬停**：放大 + 高亮（参数待细调）
- **拖拽**：全部卡片支持拖拽到外部，不允许应用内位置调整
- **右键**：暂不考虑

### 2. 子任务系统识别

#### 2.1 核心发现

产品总监说明：任务卡片的"1/3"是**子任务系统**

**关键特性**：
- 任务采用**树状结构**（父任务 → 子任务）
- 子任务有专门的名称（非简单序号）
- 子任务最终导入同一文件夹
- 制作时间不同步
- 素材不与具体子任务关联，只统计总体上传状态

#### 2.2 创建与管理

**创建位置**：在项目页的任务列表中创建

**任务模板系统**：
- 系统提供默认任务模板
- 用户可通过"模板管理"功能自定义
- 支持添加/删除/编辑任务和子任务

**默认模板**：待后续补充具体内容

#### 2.3 进度判定逻辑

**无子任务时**（叶子任务）：
- 页面刷新时检测所有素材都上传 → 自动标记完成

**有子任务时**（父任务）：
- 页面刷新时检测到所有素材上传 → 询问用户"哪个子任务完成了？"
- 用户手动勾选对应子任务 → 状态更新（如：1/3 → 2/3）
- 条件从满足变为不满足，且已有≥2个子任务被勾选 → 询问用户"是否更新状态？"

**触发时机**：页面刷新时检测

## 8. 澄清：任务页面设计

#### 8.1 重要澄清

**产品总监纠正**：
> "提醒你，任务页面不分父任务与子任务页面，一个任务就是所有的都在这个任务的页面，子任务我们会做成一个按钮放在快捷功能区，这个按钮可以显示子任务完成了多少，点开弹出弹窗也能手动更改完成情况，平时就按照之前说的，根据页内的素材上传情况来判断进度和弹窗让用户填写完成情况。"

#### 8.2 之前的错误理解

**Tech Lead 之前的理解**（错误）：
- 点击父任务 → 进入父任务的任务页（混合所有子任务素材）
- 点击子任务 → 进入子任务的任务页（独立素材）

**问题**：误以为父任务和子任务有独立的页面

#### 8.3 正确的设计

**任务页面架构**：
- ✅ 任务页面不分父任务和子任务
- ✅ 一个任务只有一个任务页面
- ✅ 所有子任务的素材都在同一个任务页面里混合显示
- ✅ 子任务只是进度追踪工具，不是独立页面

**子任务进度按钮**：
- **位置**：任务页的快捷功能区
- **显示**：`[子任务 1/3]`（显示已完成/总数）
- **点击**：弹出弹窗，显示所有子任务列表，可手动勾选完成状态
- **自动判定**：页面刷新时检测素材上传情况，如果全部上传则弹窗询问"哪个子任务完成了？"

#### 8.4 设计合理性

这个设计符合业务逻辑：
- 所有子任务的素材最终导入同一个文件夹
- 素材不与具体子任务关联
- 子任务只用于追踪制作进度
- 避免了页面层级过深

#### 11.1 背景

**产品总监说明**：

**之前的工作流程**：
1. 用户点击[转换]，程序按比例分组读取文件（50%组、70%组、30%组）

7. 重复上述过程处理下一组

**痛点**：
- 分批次转换，重复操作多
- 手动步骤繁琐（复制地址、手动输出、手动关闭窗口、手动确认）

## 12. 多选模式和上传功能设计

#### 12.1 用户需求

**产品总监说明**：
> "关于上传，我们似乎没有考虑怎么确定已经上传。我这样设想：我们会在主内容区的标题行添加多选功能，开启后追加全选按钮，总之多选之后，我们拖拽卡片然后拖拽进入浏览器上传。然后应用会弹窗问我们是上传了吗？确认之后就把刚才选中的素材复制到nextcloud的对应任务文件夹。之前设置了识别机制因此那些素材会自动更新标记为已上传。"

#### 12.2 设计方案

**核心思路**：多选 + 拖拽上传 + 确认弹窗 + 自动状态更新

**工作流程**：
1. 用户点击小标题栏的 `[多选]` 按钮进入多选模式
2. 小标题栏切换显示：`[全选] [取消全选] 已选中: X [完成多选]`
3. 所有素材卡片显示复选框，点击卡片切换选中状态
4. 用户选中需要上传的素材（可以全选或部分选择）
5. 拖拽选中的卡片到浏览器窗口，上传文件
6. 应用检测到拖拽操作，弹出确认窗口："是否已成功上传到网盘？"
7. 用户点击 `[是，已上传]`
8. 程序自动复制文件到nextcloud对应任务文件夹
9. 素材进度追踪系统自动更新状态为"已上传"

#### 12.3 界面设计

**小标题栏**：

正常模式：
```
┌─────────────────────────────────────────────────────┐
│  素材列表  [🌲树形视图] [📄名称视图] [🔄刷新] [多选]  │
└─────────────────────────────────────────────────────┘
```

多选模式：
```
┌──────────────────────────────────────────────────────────────┐
│  素材列表  [🌲树形视图] [📄名称视图] [🔄刷新] [多选✓] [全选]  │
└──────────────────────────────────────────────────────────────┘
```

点击全选后：
```
┌──────────────────────────────────────────────────────────────────┐
│  素材列表  [🌲树形视图] [📄名称视图] [🔄刷新] [多选✓] [取消全选]  │
└──────────────────────────────────────────────────────────────────┘
```

**设计说明**：
- 多选模式下，视图切换和刷新按钮保持显示
- `[多选]` 按钮变为点亮状态 `[多选✓]`
- 新增 `[全选]` 按钮，点击后文字切换为 `[取消全选]`
- 不显示"已选中计数"
- 不需要"完成多选"按钮
- 点击 `[多选✓]` 退出多选模式

#### 12.4 产品总监修正

**产品总监纠正**：
> "需要提醒的是小标题栏：多选模式只要新加一个全选按钮，点击后变成取消全选，而多选按钮进入点亮状态，再次点击取消全选模式，视图切换，刷新按钮依然保持。别的功能按钮不需要。"

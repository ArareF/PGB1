### 1. 转换功能升级讨论

#### 背景
产品总监提出转换功能存在的问题和改进需求：
1. **漏洞**：没有排除已转换的素材，可能重复转换
2. **用户控制**：需要让用户选择哪些素材要转换
3. **自动化**：静帧转换因为有映射表，可以实现自动完成
4. **扩展支持**：需要支持序列帧转精灵图的流程

#### 产品总监原话
> "哦对哦这里有个漏洞，我们要排除那些已经被标记为已经有webp版本的素材。也就是至少02_done已经有了的。"
> "然后我想这样，我们点击这个功能后，出一个弹窗，这个弹窗里会有所有没有转换版的静帧和序列帧的卡片。"
> "静帧和序列帧分开，默认多选状态等待用户选择，下面有全选功能，静帧和序列帧的小标题旁边有一键全选该类型的钩子"
> "选好后开始制作。"
> "然后因为我们这里有映射表，所以我们应该也是可以知道有多少个静帧要制作的，所以静帧的完成转换窗口应该可以在计数完成后自动启动关闭程序。"

#### 设计方案

**阶段1：素材选择弹窗**

│ ─────────────────────────────────────────────────────────── │

│  ☑ [卡片6]  ☑ [卡片7]  ☑ [卡片8]  ☑ [卡片9]  ☑ [卡片10]   │
│  ☑ [卡片11] ☑ [卡片12] ☑ [卡片13] ☑ [卡片14] ☑ [卡片15]  │
│                                                              │

│ ─────────────────────────────────────────────────────────── │
│                                                              │

**关键特性**：
1. **排除已转换素材**：
   - 静帧：检查 `02_done/[img-XX]/` 是否已存在同名 `.webp` 文件
   - 序列帧：检查 `02_done/[an-XX-YY]/` 是否已存在对应的精灵图三件套
   - 存在则跳过，不显示在弹窗中

2. **默认多选状态**：所有未转换的素材默认勾选

3. **分类全选钩子**：
   - **[全选]**：全局全选按钮（选中所有静帧+序列帧）
   - **[全选静帧]**：只选中静帧区的所有卡片
   - **[全选序列帧]**：只选中序列帧区的所有卡片

**阶段2：静帧转换（imagine工作流）**

**关键升级**：

2. **自动完成检测**：
   - 当 `已整理数 == 总数` 时
   - 等待2秒（确保所有文件都已处理）
   - 自动执行清理操作（关闭imagine、关闭窗口、关闭悬浮窗）
   - 无需用户点击 `[完成转换]` 按钮
3. **手动完成**：用户也可以随时点击 `[完成转换]` 提前结束

**阶段3：序列帧转换（TexturePacker工作流）**

待明确的问题：

- [ ] 帧率来源（用户输入 vs 自动读取）
- [ ] 批量处理方式
- [ ] 输出监控机制

#### 决策要点
- ✅ 转换功能分为三个阶段：选择弹窗 → 静帧转换 → 序列帧转换
- ✅ 静帧转换支持自动完成（因为有映射表知道总数）
- ✅ 排除已转换素材（避免重复转换）
- ✅ 默认全选，让用户取消不需要的
- ✅ 分类全选钩子（提高用户控制灵活性）

### 2. 素材卡片预览图数据源规则

#### 背景
产品总监提醒：

> "需要注意的是我们的任务素材卡片的预览图要始终使用original中的素材"

#### 设计规则

**核心规则**：任务页素材卡片的预览图必须始终使用 `00_original/` 中的素材

**原因**：
- 确保预览图展示的是原始质量，而非缩放或转换后的版本
- 用户在任务页看到的预览应该是最准确的原始素材

**技术实现**：
- 卡片渲染时，优先从 `00_original/` 读取文件
- 而不是从 `01_scale/` 或 `02_done/` 读取

**更新位置**：
- `design/卡片设计.md` - 预览图生成规则章节
- 添加了数据源规则说明和对照表

#### 决策要点
- ✅ 所有任务页素材卡片预览图使用 `00_original/` 素材
- ✅ 包括静帧、序列帧、精灵图、视频等所有类型
- ✅ 确保预览质量始终为原始质量

### 3. 转换功能升级 - 序列帧支持

#### 背景
产品总监提供了TexturePacker自动化脚本，说明序列帧也可以高度自动化处理。

#### 产品总监原话
> "好，我要再修改一下悬浮窗的设计，我们搞到了序列帧的快捷方法，所以我们把序列帧也加进悬浮窗启动关闭的条件中。"
> "我们选中序列帧后，开始转换，texturepacker可以高度自动化，一会儿我给你看代码。"
> "序列帧也会输出到02_done这个文件夹中，此时序列帧会有webp,tps,plist三个文件，这三个文件也挪动到对应的[an-大小-帧率]文件夹中。"

关于帧率输入方案：
> "好，那就方案B，就在下方增加这个输入框吧，然后没填完也就灰着锁定。"

#### PowerShell脚本分析

产品总监提供的TexturePacker自动化脚本：

```powershell
$tp  = "C:\Program Files\CodeAndWeb\TexturePacker\bin\TexturePacker.exe"
$gui = "C:\Program Files\CodeAndWeb\TexturePacker\bin\TexturePackerGUI.exe"
$outputDir = "C:\path\to\output"

# 逐个处理（推荐方案）
foreach ($f in $folders) {
    $name = Split-Path $f -Leaf
    & $tp $f ... --save "$outputDir\$name.tps"
    $proc = Start-Process $gui "$outputDir\$name.tps" -PassThru
    $proc.WaitForExit()
    $scale = 从 .tps 解析 scale
}
```

关键点：
- **CLI生成.tps**：用命令行版本TexturePacker生成工程文件
- **GUI调整参数**：打开GUI让用户微调（scale、quality等）
- **逐个等待**：处理完一个再处理下一个（不是批量等待）
- **解析scale**：从.tps文件（XML格式）解析最终scale值
- **三件套整理**：.webp、.plist、.tps移动到 `[an-{scale}-{fps}]/`

#### 设计方案

**1. 选择弹窗添加帧率输入**

序列帧卡片下方显示帧率输入框：
```
┌─────────────────────┐
│  ☑                  │
│   [序列帧预览图]    │
│  main_vfx_loop      │
│  [70%]              │
├─────────────────────┤
│ 帧率: [24] fps      │ ← 必填输入框
└─────────────────────┘
```

**验证规则**：
- 必须是正整数（1-120范围）
- 未填写或无效时，输入框显示红色边框
- 所有选中的序列帧都填写有效帧率后，"开始制作"按钮才可点击
- 按钮置灰时显示提示："请为所有序列帧填写帧率"

**2. 悬浮窗统一追踪进度**

```
┌────────────────────────────────────────┐
│ 转换进行中...                   [×]    │
│ ──────────────────────────────────────│
│ 已经复制地址，等待用户操作             │
│                                        │
│ 目标地址：                             │
│ D:\...\03_Render_VFX\...\02_done       │
│                          [📋 再复制]   │
│ ──────────────────────────────────────│
│ 静帧: 0 / 15 个文件                    │
│ 序列帧: 0 / 8 个文件                   │
│                                        │
│              [完成转换]                │
└────────────────────────────────────────┘
```

**执行顺序**：
1. 先处理静帧（imagine工作流）
2. 静帧完成后，处理序列帧（TexturePacker工作流，逐个等待）
3. 全部完成后自动关闭程序

**自动完成条件**：
- 只有静帧：静帧全部完成
- 只有序列帧：序列帧全部完成
- 两者都有：静帧 AND 序列帧全部完成

**3. 序列帧转换详细流程**

逐个处理每个序列帧文件夹：

1. **生成.tps工程文件**（CLI）：
   ```bash
   TexturePacker.exe 序列帧文件夹 \
     --sheet 输出.webp \
     --data 输出.plist \
     --format cocos2d-x \
     --texture-format webp \
     --scale 0.5 \
     --save 输出.tps
   ```

2. **打开GUI让用户调整**：
   - 启动 `TexturePackerGUI.exe 输出.tps`
   - 用户在GUI中调整参数（scale、quality等）
   - 用户点击"Publish"保存并关闭GUI

3. **等待GUI关闭**：
   - `process.WaitForExit()`
   - 阻塞直到用户关闭GUI

4. **解析最终scale值**：
   - 从.tps文件（XML格式）解析 `<key>scale</key><double>0.7</double>`
   - 转换为百分比：0.7 → 70

5. **整理三件套**：
   - 移动 `.webp`、`.plist`、`.tps` 到 `[an-{scale}-{fps}]/`
   - 帧率来自用户在选择弹窗中输入的值

6. **更新进度**：
   - 悬浮窗显示 `序列帧: X / Y 个文件`
   - 继续处理下一个序列帧

**技术要点**：
- **.tps文件解析**：XML格式，提取 `<scale>` 节点值
- **三件套完整性检测**：移动前检查三个文件都存在
- **进程等待**：`WaitForExit()` 同步等待用户操作
- **文件夹命名**：`[an-{scale}-{fps}]`，scale从.tps解析，fps从用户输入

#### 决策要点
- ✅ 序列帧帧率输入集成到选择弹窗中（方案B）
- ✅ 未填写完整帧率时，"开始制作"按钮置灰
- ✅ 悬浮窗同时追踪静帧和序列帧进度
- ✅ 逐个处理序列帧（不是批量等待）
- ✅ 从.tps文件解析最终scale值（用户可在GUI中调整）
- ✅ 全部完成后自动关闭程序（静帧+序列帧都完成）

### 4. 外部工具路径配置和窗口管理

#### 背景
产品总监提醒：

> "texturepacker的地址我刚才好像给的是硬编码，imagine和它的安装地址后面我们都会通过 应用设置 里提供的。"
> "然后就是因为会打开很多窗口，所以我希望也能够自动把imagine和texturepacker的gui窗口摆到前面，当然imageine在最前面。"

#### 设计方案

**1. 应用设置添加工作流配置**

在应用设置页面的"工作流设置"分类中添加：

```
▶ 工作流设置
───────────────────────────────────────────
imagine 路径: [C:\...\imagine.exe]  [浏览...]
TexturePacker CLI: [C:\...\TexturePacker.exe]  [浏览...]
TexturePacker GUI: [C:\...\TexturePackerGUI.exe]  [浏览...]
```

**配置项说明**：

1. **imagine 路径** (`workflow.imaginePath`)
   - 默认值：`C:\Program Files\imagine\imagine.exe`
   - 用途：静帧格式转换工具
   - 验证：启动时检查文件是否存在

2. **TexturePacker CLI 路径** (`workflow.texturePackerCliPath`)
   - 默认值：`C:\Program Files\CodeAndWeb\TexturePacker\bin\TexturePacker.exe`
   - 用途：序列帧打包命令行工具

3. **TexturePacker GUI 路径** (`workflow.texturePackerGuiPath`)
   - 默认值：`C:\Program Files\CodeAndWeb\TexturePacker\bin\TexturePackerGUI.exe`
   - 用途：序列帧打包GUI工具

**自动检测逻辑**：
- 首次启动时，程序尝试在常见安装路径查找这些工具
- 如果找到，自动填充到配置中
- 如果找不到，留空并显示提示："请手动配置工具路径"

**配置文件结构**：
```json
{
  "workflow": {
    "imaginePath": "C:\\Program Files\\imagine\\imagine.exe",
    "texturePackerCliPath": "C:\\Program Files\\CodeAndWeb\\TexturePacker\\bin\\TexturePacker.exe",
    "texturePackerGuiPath": "C:\\Program Files\\CodeAndWeb\\TexturePacker\\bin\\TexturePackerGUI.exe"
  }
}
```

**2. 窗口管理（自动置前）**

因为转换过程中会打开多个窗口，需要自动将它们置于前面方便用户操作。

**优先级**：imagine > TexturePacker GUI

**技术实现**（Windows API）：

1. **imagine窗口**：
   - 启动imagine后立即调用 `bringWindowToFront(imaginePid)`
   - 将imagine窗口置于最前面（优先级最高）

2. **TexturePacker GUI窗口**（逐个处理）：
   - 每次启动新的TexturePacker GUI后调用 `bringWindowToFront(guiPid)`
   - 因为是逐个处理，每次只有一个GUI在前面
   - 用户关闭当前GUI后，下一个GUI自动打开并置前

**实现步骤**：

```rust
// 1. 等待窗口创建（轮询检测，最多2秒）
await wait_for_window(pid)

// 2. 将窗口置于最前面
unsafe {
    // 恢复窗口（如果最小化）
    ShowWindow(hwnd, SW_RESTORE);

    // 临时置顶
    SetWindowPos(hwnd, HWND_TOPMOST, 0, 0, 0, 0, SWP_NOMOVE | SWP_NOSIZE);

    // 取消置顶但保持在最前（避免永久置顶）
    SetWindowPos(hwnd, HWND_NOTOPMOST, 0, 0, 0, 0, SWP_NOMOVE | SWP_NOSIZE | SWP_SHOWWINDOW);

    // 激活窗口
    SetForegroundWindow(hwnd);
}
```

**注意事项**：
- 不使用永久置顶，避免遮挡悬浮窗和其他重要窗口
- 使用临时置顶后立即取消，只是将窗口移到Z-order顶部
- `SetForegroundWindow` 可能受Windows安全限制，需要程序有焦点时才能生效

#### 决策要点
- ✅ 外部工具路径通过应用设置配置（不硬编码）
- ✅ 首次启动时自动检测常见安装路径
- ✅ 启动imagine和TexturePacker GUI后自动置前
- ✅ imagine优先级最高（最前面）
- ✅ 使用临时置顶而非永久置顶

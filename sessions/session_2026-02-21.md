## 🔧 技术实现

### useRubberBandSelect composable

- `position:fixed` 视口坐标覆盖层 + `getBoundingClientRect` 碰撞检测
- `MOVE_THRESHOLD = 5px`：防止误触发
- `justFinished` ref：屏蔽框选后的幽灵 click 事件（`setTimeout(0)` 异步重置）
- `onContainerScroll`：滚动时立即终止框选，防止起点失效
- 关键边界处理：container 为 null 时调用 `onMouseUp()` 而非直接 return（防事件监听器泄漏）

### 两种接入模式

| 模式 | 页面 | isEnabled |
|------|------|-----------|
| 有开关（多选/全选按钮） | TaskPage / GameIntroPage / MaterialsPage | `isMultiSelect` ref |
| 始终开启 | ScalePage / ConvertPage | `ref(true)` |

### 冲突解决

| 冲突点 | 解决方案 |
|--------|---------|
| 卡片 mousedown 拖出 vs 框选 | `onContainerMouseDown` 检测 `e.target.closest(cardSelector)`，点在卡片上直接跳过 |
| 框选结束 click 关闭侧边栏 | `justFinished` 保护 + `setTimeout(0)` 重置 |
| 未开多选时误触发 | `isEnabled = isMultiSelect`，为 false 时直接返回 |
| 滚动时起点失效 | 容器 scroll 事件触发 `onMouseUp()` |

## 🐛 Bug 修复 #2：Prototype 素材大小分区失效

**现象**：Prototype 任务的素材页（TaskPage 树形视图）只按子分类（symbol / big_win / …）分组，缩放后的素材无法按大小（[100] / [70] / 原始）区分显示。

**根因一（前端）**：`TaskPage.vue` `groupedMaterials` 的 Prototype 分支仅做子分类分组，完全忽略 `MaterialInfo.scales` 字段，无大小子分组逻辑。

**根因二（后端）**：`scan_materials_prototype` 中序列帧 scales 调用了 `collect_scales_for_sequence`，该函数只在 `[an-XX-YY]/base_name.*` 层级查找，而 Prototype 的实际路径是 `[an-XX-YY]/{sub_name}/base_name.*`，多一层 sub_name，导致序列帧 scales 永远返回空。

**关于目录冲突**（用户假设）：`apply_task_changes` 在 `01_scale/` 下创建平铺占位目录（`01_scale/symbol/`），`execute_scaling` 在 `01_scale/[100]/symbol/` 写入真实文件，两者**不冲突**——`collect_scales_for_proto_image` 过滤规则 `starts_with('[')` 已正确跳过平铺目录。

**修复**：

| 文件 | 改动 |
|------|------|
| `src/views/TaskPage.vue` | `groupedMaterials` Prototype 分支：改为先按子分类分组，再对每个子分类做 scale 子分组（原始 / [100] / …），结构与普通任务对称 |
| `src-tauri/src/commands.rs` | 新增 `collect_scales_for_proto_sequence(base_name, sub_name, done_dir)`，在 `[an-XX-YY]/{sub_name}/` 下查找；`scan_materials_prototype` 序列帧改用此函数 |

## 🐛 Bug 修复 #3：Prototype 静帧转换后 webp 不识别、不转移

**现象**：Prototype 任务执行静帧转换后，Imagine 输出的 .webp 文件无法被自动识别，也不会被移动到 `02_done/`。

**根因**：`conversion.rs` 的 `handle_file_event` 检查 `parent.parent() != scale_dir` 来确认文件是否在 `01_scale/[XX]/` 层级。

- 普通任务路径：`01_scale/[100]/file.webp` → `parent.parent() == scale_dir` ✓
- Prototype 路径：`01_scale/[100]/symbol/file.webp` → `parent.parent() == 01_scale/[100]/` ✗ → 全部被 `continue` 跳过

**修复**（`src-tauri/src/conversion.rs`）：在原有普通任务分支之外增加 Prototype 分支——检测 `grandparent.parent() == scale_dir`，提取 `scale_parent`（`[XX]` 层）和 `subcat`（子分类名），目标目录相应改为 `02_done/[img-XX]/{subcat}/`，与 `find_file_in_proto_subdirs` 的路径预期完全一致。

| 情形 | 源路径 | 目标路径 |
|------|--------|---------|
| 普通任务 | `01_scale/[100]/fire_01.webp` | `02_done/[img-100]/fire_01.webp` |
| Prototype | `01_scale/[100]/symbol/fire_01.webp` | `02_done/[img-100]/symbol/fire_01.webp` |

## ✨ 新功能：游戏介绍页「启动原型」按钮

**背景**：设计文档 `界面设计.md` L681-691 规定，当 `00_Game Design & Doc` 目录下存在 Unity 游戏构建时，在顶部导航快捷功能区显示 `[启动原型]` 按钮。

### 改动文件

| 文件 | 类型 | 说明 |
|------|------|------|
| `src-tauri/src/commands.rs` | 修改 | 新增 `find_unity_game_exe` 命令 |
| `src-tauri/src/lib.rs` | 修改 | 注册新命令 |
| `src/views/GameIntroPage.vue` | 修改 | 静态 setNavigation 重构为 refreshNav()，新增 scanUnityGameExe() |

### 技术实现

**检测策略**：不依赖文件夹名称（`Build/` 等），而是以 `UnityCrashHandler64.exe` 为 Unity 构建的万能指纹。Unity 引擎在构建时必然将该 crash handler 与主 exe 放在同一目录。

**`find_unity_game_exe` Rust 命令**：
- 递归遍历 `00_Game Design & Doc` 全目录树（`std::fs::read_dir` + 递归 walk）
- 逐层检测：当前目录是否含 `UnityCrashHandler64.exe`（大小写不敏感）
- 找到后，返回同目录中第一个其他 `.exe` 文件的路径
- 目录不存在 / 无权限 → 返回 `None`，不报错

**前端流程**：`onMounted` → `loadFiles` → `scanUnityGameExe` → 有结果则 `refreshNav()`（插入启动按钮）

**启动**：复用现有 `open_file` 命令（ShellExecuteW）

### 设计决策

- **不用 `Build/` 硬编码**：目录名因项目而异，甚至可能嵌套多层；`UnityCrashHandler64.exe` 是更可靠的特征
- **单次扫描**：仅在页面 mount 时扫描一次，不在每次刷新时重扫（exe 变更场景极少，可重新进页面触发）
- **复用 `open_file`**：ShellExecuteW 已被充分测试，无需新增启动逻辑
